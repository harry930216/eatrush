> **來源**：2026-07-13 三專案文件系統對照（UcMarket / EatRush / vocabulary-v5，完整拆解在 vocabulary-v5 repo 的 `docs/流程手冊/91-同儕範本拆解-UcMarket.md`）。結論：EatRush 的文件在決策密度上不輸任何人，缺的全是「**對外可讀性**」——門面、現況、引用一致性。三個洞都有現成範本可抄。
> **規則**：Harry 自己修（照 AGENTS.md 精神——文件也是練習），AI 只提供本清單與修後 review。修完打勾＋附一句結論。
> **用法**：同 `eatrush-review-todo.md` 的 loop 規則——本檔可勾可髒。不用一次修完；洞 1 最值得先做（面試官正在看的就是它）。
> **退役條件**：三洞全補即功成身退；之後「文件與磁碟是否一致」由洞 3 建立的檢查習慣接手，不回這裡排隊。

## 洞 1：根目錄沒有 README.md（門面缺失）

- [ ] 完成（結論：　　　　　　）

**現況**：clone 下來第一眼是 `AGENTS.md` 和 `pom.xml`。`docs/eatrush-readme-template.md` 寫得很好，但它排 Step 8/10 才定稿——定稿前整個 repo 處於「無門面」狀態。

**為什麼是洞**：template 自己的設計原則就是答案——「面試官平均停留 3 分鐘，第一屏要讓他決定值得往下」。無門面期間，每個看過 repo 的人都在白白流失。UcMarket 給人「條理分明」的第一印象，有一半是 README＋docsREADME 兩張臉貢獻的。

**修法方向**（不代寫，給座標）：
1. 不等 Step 8——先放「**施工中版 README**」：從 template 抄骨架，只填現在填得出的段（一句話定位、30 秒版、文件地圖）。
2. 加一節「目前進度」＝洞 2 的迷你現況基準（見下）。
3. 還填不了的段落（demo GIF、測試帳號表、故意做壞實驗的截圖）保留【】並標「Step N 補」——**施工中的誠實比完工的假象值錢**，這本身就是給面試官看的工程態度。

**參考範本**：UcMarket `README.md:3` 的開頭引言寫法（一行講清楚「規劃不代表已完成、現況看哪份」）；你自己的 template 第 3 行設計原則。

**驗收**：找一個沒看過專案的人（或叫 AI 扮演面試官）只讀 README，能答出四題——這專案是什麼？怎麼跑起來？現在做到哪？其他文件在哪？

## 洞 2：沒有現況基準（「做到哪」要考古）

- [ ] 完成（結論：　　　　　　）

**現況**：進度散在 playbook 各 Step 的驗證 checkbox 和最新一篇 dev-log 裡。要回答「現在什麼能跑」得考古兩份文件＋git log。

**為什麼是洞**：文件會過期說謊，而且規劃類文件（spec/playbook）天生描述未來。UcMarket 的解法是 `docs/current-implementation.md`（僅 68 行）：基準 commit＋已接通／尚未實作＋驗證指令＋一句衝突裁決（「若與歷史文件衝突，以程式碼、測試與本文件為準」）。vocabulary-v5 已照抄落地，v5 版第一次盤點就抓到一條前端按鈕打不存在端點的 404 斷鏈——**寫這份文件的過程本身就是抓蟲**。

**修法方向**：
1. 單人專案劑量可以小——**先併進 README 的「目前進度」節即可**，不必獨立成檔（抄招式，不抄劑量；等 repo 變大再拆出去）。
2. 關鍵四件，缺一不可：基準 commit／已完成到哪個 Step（含一行「尚未實作」）／驗證指令（照打就能確認活著）／衝突裁決一句話。
3. **維護義務**：每個 Step 出口 commit 後順手更新基準——過期的錨比沒有更糟，因為它披著權威的皮說謊。可以把「更新現況節」加進 playbook 迴圈規則 6 的出口儀式裡。

**驗收**：文件寫的基準 commit＝`git log -1` 的 hash；照文件的驗證指令跑一遍全綠。

## 洞 3：幽靈引用（宣稱的目錄不存在）

- [ ] 完成（結論：　　　　　　）

**現況**：`AGENTS.md` 文件地圖列了 `docs/troubleshooting/` 與 `docs/iterations/`；playbook 自救梯收尾（迴圈規則 5）也叫你把排錯筆記寫進 `docs/troubleshooting/`——**兩個目錄實際都不存在**。Flyway 換機那篇排錯筆記（本質正是 troubleshooting）寫進了 `docs/dev-log/`。

**為什麼是洞**：讀者（含未來接手的 AI）照地圖找檔會踩空，而且「地圖說 A、實際做 B」會慢慢腐蝕整套文件的可信度。UcMarket 的 docsREADME 維護原則有一條專門防這個：「新增圖檔時，確認 Markdown 裡引用的檔名真的存在」。

**修法方向**（二選一，都合法，重點是文件與磁碟一致）：
- a) 建 `docs/troubleshooting/`，把 `dev-log/2026-07-09-flyway-second-machine-stale-build.md` 搬過去（它自己都說「不對應任何施工 step」）；`iterations/` 先建空目錄放 `.gitkeep`。
- b) 改 `AGENTS.md` 地圖與 playbook 規則 5：註明「排錯筆記暫併 `dev-log/`；`iterations/` v1.1 起才建」。

**順手觀察項**（非洞，看到順便講）：commit 訊息有重複連發（同訊息兩條）與「文件更新2」式流水帳——playbook 出口儀式其實已有標準（「訊息講得出這段在防什麼」），照自己立的規矩走就好。

**驗收**：把 `AGENTS.md` 與 playbook 裡出現的所有 `docs/` 相對路徑列出來，逐一確認檔案或目錄真的存在，零踩空。
