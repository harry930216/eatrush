## 解開的兩層(兩個獨立決策,混講必卡)

### Q1 能力題:矩陣 vs level 比大小 — 看權限關係的形狀

- **鏈(chain,全序)**=「越往上越開放」→ 能壓成一個數字,level 比大小。§4.1 方案權益就是:商業承諾貴方案吃到所有便宜的,永不破
- **偏序(partial order)**=職能互不包含 → 一維數線裝不下,只有矩陣(逐格打勾)裝得下
- 殺手反例(我自己提的「會計」):會計看報表、不碰訂單 — level 推演:會計須 <1(不能管訂單)、報表門檻須 ≤會計 level → 門檻 ≤0 → **店員全看得到財務** ✗;反向設 level 3 → 會計自動拿到改菜單改庫存 ✗。**無解,因為會計 vs 店員本來就不可比(incomparable)**
- 誠實面:V1 的 STAFF⊂OWNER 恰好是鏈,level 當下真的解得了 — 矩陣是對已知變化點(§15-13 儀表板圈的 REPORT_VIEW)的**低成本預付**(一張表+seed 四列+一句 JOIN);反向遷移(level→矩陣)=所有 @PreAuthorize 重寫,代價不對稱

### Q2 形式題:2 表 vs 3 表 — role 這個鍵用什麼形式(與 Q1 完全獨立)

- 兩版**都是矩陣、能力完全等價**;差別只是 role 鍵用**自然鍵(natural key,'OWNER' 字串)還是代理鍵(surrogate key,id 數字)** — 3 表版就是把字串換成指向 role 表的數字
- 代理鍵三個存在理由,role 逐條不中:①自然鍵會變?enum 常數改名=全 code 重構級事件 ②太笨重?五個字元 ③有屬性要掛?零屬性(對照:permission 有 name 要掛才配表)
- **正規化錯覺**:「'OWNER' 重複三列該抽表」— 錯。正規化消除的是**屬性值**重複(防 update anomaly),**鍵值**重複不歸它管(member_id 重複萬次沒人喊消除)。字串重複看起來像 redundancy、數字重複不像,純視覺錯覺
- 對照組在自己系統裡:'STOCK_MANAGE' 也是字串進 @PreAuthorize/JWT,從沒不安過 — code 欄天生就是「設計來當識別子的穩定字串」
- 真相源(single source of truth)視角:role 集合**封閉** → 真相源在 code(Java enum),DB 字串是引用;permission **開放** → 真相源在 DB(表)

## 業界對照

- 光譜四檔:enum 硬編碼(內部工具/新創,量大)→ 表化 RBAC 3~5 表(中型 SaaS)→ IAM 外包(Keycloak/Auth0)→ 政策引擎(ABAC/ReBAC,Zanzibar)
- **Spring Security 只認 GrantedAuthority 字串集合 = 2 表世界觀**,框架從不要求 role 是實體
- Keycloak role 全表化 = 通用平台不知你的業務,角色行為只能全靠資料拼裝
- 3 表殺手場景:**多租戶 SaaS**(每家店自訂「跑菜員/會計」,不可能改 enum 重部署);成立前提=**代碼零 hasRole 寫死**,角色退化為權限包(permission bundle)的名字,否則 INSERT 出來的新角色是空殼
- 「一人多角」(老闆兼會計)逼出的不是 role 表,是 **user_role 中間表**(業界完整 RBAC 五表:user/role/permission + 兩張中間表);V1 member.role 單值欄=一人一角的刻意簡化

## 第二戰(同日晚,寫 DDL 時):我宣布「改 3 表」,六輪後投降

寫 role_permission 的 DDL 時我決定改 3 表(動機:3 表我看得懂 + 觀念 5 那句「裸字串打錯字靜默失權」不想擔),要 AI 認同或反駁。戰報:

### AI 三發反駁(第一發最重)

1. **型別安全位置交換**:「判斷角色」的 Java 句子躲不掉(註冊要設、業務要分流)。2 表版 Java 用 enum:`me.getRole() == Role.CUSTOMER` — 打錯**編譯不過,IDE 紅線**。3 表版 member 存數字:`roleId == 1`(1 是誰?seed 順序一變全錯)、或查回字串 `.equals("CUSTOMR")`(**編譯過、runtime 才錯、永遠 false 靜默壞**)、或回頭建 enum = **兩份清單人肉同步**。而且 3 表版的字串沒消失只是搬家 — `findRoleIdByCode("CUSTOMER")` 的參數還是字串,typo 風險原地不動,還多一次查表
2. **FK 防不了我真正怕的錯**:FK 只擋「不存在的 id」;seed 把 role_id 該填 2 填成 3,完全合法、完全靜默 — 數字錯比字串錯更難肉眼抓('ONWER' 一眼假,2 和 3 都長得對)。我要買的「防打錯」,FK 沒有賣
3. CHECK 約束 3 行 = DB 層拒絕爛字串(psql 玩具實驗親手摸:裸欄位靜默吞下 'ONWER' vs CHECK 版 INSERT 當場 ERROR — 「靜默失權」與解藥各摸一次)

### 我的反擊戰果:打掉 CHECK

我反打「**你抹除了擴充性**」— 真擊中:CHECK = 集合定義在 DB 的第二份複本(AI 第二輪自己說過「抄回 DB 一份會漂移」,第四輪又端出來 = 自打臉),加角色的 migration 得寫 ALTER 不是 INSERT。AI 認錯收回 CHECK,退到**純 2 表(裸字串)**。
但同一把刀也砍中 3 表:同步點數 — 純 2 表 = 2(enum+seed)、CHECK 版 = 3、**3 表 = 3**(enum + role 表 + seed)。我用來殺 CHECK 的理由,殺不出 3 表的優勢。

### 擴充性總帳(把「擴充性」這個詞拆開算)

加一個角色的全程:①行為進 Java ②進清單 ③配權限 seed ④重部署。**瓶頸 = ①④,任何方案逃不掉**(V1 的角色有寫死的業務行為);3 表只把②的 ALTER 換成 INSERT = **優化非瓶頸段**(車速卡在引擎,換了個更順的門把)。
真擴充性 = 「不改 code 不部署,調整誰有什麼權限」— **矩陣本身已經給了**(改 role_permission 列 + 重登生效),2 表就有。3 表獨賣的「不改 code 生新角色」前提是代碼零角色感知,被 V1 架構封印 — 付了錢提不走貨。

### 終局驗收:實戰 trace「加會計」

enum 加一字 + 一支 **INSERT-only** migration(permission / role_permission / member 各一句)+ 部署。role 欄是 varchar — **新字串直接放進來,它從來就沒關上**。兩個發現:
- 角色的新增成本由「**矩陣裝不下的專屬行為量**」決定:會計 = 純矩陣角色(0 專屬行為)≈ 配置級變更;CUSTOMER 當初有 plan/點餐行為 = 貴 — **與表數無關**
- 一人多角(老闆兼會計)逼出的是 **user_role 中間表**(五表模型),不是 role 表;且 user_role 的 role 欄照樣可存字串 — 連五表模型都不需要 role 實體

### 過程的價值(自己記下)

我的兩次反擊都真實改變了 AI 的立場:第一戰逼出 Q1/Q2 分層、第二戰打掉 CHECK — 但兩次都不是把它推向 3 表,而是推向**更純的 2 表**。辯輸了,兩拳都沒白打。面試講這個攻防過程,比背結論值錢。

## 一句話帶走

**角色由誰定義 — 工程師定義(行為在 code)= enum + 字串(2 表);營運者定義(角色是可配置資料)= 表 + 代理鍵(3 表)。**
EatRush 單店、角色我定 → 2 表。場景翻成多租戶 SaaS,答案就翻。
第二戰追加:**typo 防護要放在錯誤最常發生的地方(Java 業務代碼 → enum 編譯期紅線),不是放在圖上最好看的地方(DB 約束)。**
